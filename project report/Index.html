Abstract

This project presents Medium Clone, a full-stack web application designed to replicate the core functionality of the popular blogging platform Medium. The application demonstrates the practical implementation of database management concepts including relational database design, normalization (BCNF), indexing strategies, transaction management with ACID compliance, and complex SQL query optimization.

The system is built using PostgreSQL as the primary database, Node.js and Express.js for the backend RESTful API, and React.js for the frontend user interface. The database schema consists of six normalized tables: Users, Articles, Comments, Likes, Bookmarks, and Followers, with carefully designed relationships and constraints.

Key features implemented include user authentication with JWT, article creation with rich text editing, social interactions (follow/unfollow, likes, bookmarks), nested comments, full-text search functionality, and image upload capabilities. The application successfully demonstrates the application of ACID properties through explicit transaction management and achieves optimal query performance through strategic indexing on foreign keys and frequently queried columns.

Keywords: PostgreSQL, Database Design, Normalization, BCNF, Indexing, Transaction Management, ACID, Full-Stack Development, Blogging Platform

5. Database Design

5.1 Database Schema

The database consists of six main tables with carefully defined relationships:

5.2 Users Table

CREATE TABLE users ( id SERIAL PRIMARY KEY, username VARCHAR(50) UNIQUE NOT NULL, email VARCHAR(100) UNIQUE NOT NULL, password_hash VARCHAR(255) NOT NULL, full_name VARCHAR(100), bio TEXT, avatar_url VARCHAR(255), created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

5.3 Articles Table

CREATE TABLE articles ( id SERIAL PRIMARY KEY, title VARCHAR(255) NOT NULL, subtitle VARCHAR(255), content TEXT NOT NULL, cover_image VARCHAR(255), read_time INT DEFAULT 1, author_id INT REFERENCES users(id) ON DELETE CASCADE, tags TEXT[], status VARCHAR(20) DEFAULT 'published', views INT DEFAULT 0, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

5.4 Comments Table

CREATE TABLE comments ( id SERIAL PRIMARY KEY, content TEXT NOT NULL, article_id INT REFERENCES articles(id) ON DELETE CASCADE, user_id INT REFERENCES users(id) ON DELETE CASCADE, parent_comment_id INT REFERENCES comments(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

5.5 Likes Table

CREATE TABLE likes ( id SERIAL PRIMARY KEY, article_id INT REFERENCES articles(id) ON DELETE CASCADE, user_id INT REFERENCES users(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, UNIQUE(article_id, user_id) );

5.6 Bookmarks Table

CREATE TABLE bookmarks ( id SERIAL PRIMARY KEY, article_id INT REFERENCES articles(id) ON DELETE CASCADE, user_id INT REFERENCES users(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, UNIQUE(article_id, user_id) );

5.7 Followers Table

CREATE TABLE followers ( id SERIAL PRIMARY KEY, follower_id INT REFERENCES users(id) ON DELETE CASCADE, following_id INT REFERENCES users(id) ON DELETE CASCADE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, UNIQUE(follower_id, following_id), CHECK (follower_id != following_id) );

8. Normalization Analysis

8.1 First Normal Form (1NF) ✓

Rule: All attributes contain atomic values; no repeating groups.

Verification: All tables satisfy 1NF. The tags attribute in articles uses PostgreSQL ARRAY type which is considered atomic at the database level.

8.2 Second Normal Form (2NF) ✓

Rule: 1NF + No partial dependencies on a composite primary key.

Verification: All tables with single-column primary keys automatically satisfy 2NF. For tables with composite keys (likes, bookmarks, followers), non-key attributes depend on the full composite key.

8.3 Third Normal Form (3NF) ✓

Rule: 2NF + No transitive dependencies.

Verification: All non-prime attributes depend directly on the primary key, not on other non-prime attributes. For example, in the articles table, author information is stored in the users table, not duplicated.

8.4 Boyce-Codd Normal Form (BCNF) ✓

Rule: For every functional dependency X → Y, X must be a superkey.

Verification: All tables satisfy BCNF:

  • users: id, username, email are all superkeys
  • articles: id is a superkey
  • comments: id is a superkey
  • likes: (article_id, user_id) and id are superkeys
  • bookmarks: (article_id, user_id) and id are superkeys
  • followers: (follower_id, following_id) and id are superkeys

10. Indexing Strategy

10.1 What is Indexing?

An index is a data structure (B-Tree in PostgreSQL) that improves the speed of data retrieval operations at the cost of additional storage and write overhead.

11. Transaction Management

11.1 What is a Transaction?

A transaction is a sequence of database operations that are executed as a single unit of work. It must satisfy the ACID properties:

  • Atomicity: All operations succeed or all fail
  • Consistency: Database remains in a valid state
  • Isolation: Concurrent transactions don't interfere
  • Durability: Committed changes persist

const createArticle = async (req, res) => { const client = await pool.connect(); try { await client.query('BEGIN'); // Operation 1: Insert article const articleResult = await client.query( 'INSERT INTO articles (...) 12. SQL Queries

12.1 Complex Queries Used

Query 1: Get Article with Author and Stats

SELECT a.*, u.username as author_username, u.full_name as author_full_name, (SELECT COUNT(*) FROM likes WHERE article_id = a.id) as likes_count, (SELECT COUNT(*) FROM comments WHERE article_id = a.id) as comments_count, (SELECT EXISTS(SELECT 1 FROM likes WHERE article_id = a.id AND user_id = $2)) as is_liked FROM articles a JOIN users u ON a.author_id = u.id WHERE a.id = $1;

Query 2: Search Articles and Users

-- Search articles SELECT a.id, a.title, a.read_time, u.full_name as author_full_name FROM articles a JOIN users u ON a.author_id = u.id WHERE a.status = 'published' AND (a.title ILIKE $1 OR a.content ILIKE $1 OR $1 = ANY(a.tags)) ORDER BY a.created_at DESC LIMIT 5; -- Search users SELECT id, username, full_name, bio, avatar_url FROM users WHERE username ILIKE $1 OR full_name ILIKE $1 LIMIT 5;

Query 3: Get User Profile with Stats

SELECT u.*, (SELECT COUNT(*) FROM followers WHERE following_id = u.id) as followers_count, (SELECT COUNT(*) FROM followers WHERE follower_id = u.id) as following_count, (SELECT COUNT(*) FROM articles WHERE author_id = u.id) as articles_count, (SELECT EXISTS(SELECT 1 FROM followers WHERE follower_id = $2 AND following_id = u.id)) as is_following FROM users u WHERE u.id = $1;

Query 4: Get Feed (Articles from Followed Users)

SELECT a.*, u.username, u.full_name, (SELECT COUNT(*) FROM likes WHERE article_id = a.id) as likes_count FROM articles a JOIN users u ON a.author_id = u.id WHERE a.author_id IN ( SELECT following_id FROM followers WHERE follower_id = $1 ) ORDER BY a.created_at DESC LIMIT 20;
